# CLAUDE.md

Guidance for working in this repo.

## What this is

A pure-frontend, Townscaper-style 3D building tool for students. The product is **a spatial-language editor**: the user defines *space* (rectangular volumes per floor, stacked) and the system **automatically materializes** the building skin (floors, exterior walls with deterministic windows + one ground door, corner columns, a continuous flat roof). The whole composition exports to a single **GLB**. Assets are Kenney's Building Kit (flat-roof only — no pitched roofs).

Stack: **React 19 + React Three Fiber v9 + @react-three/drei v10 + three 0.180 + Zustand 5 + Vite + TypeScript.**

## Run / build

```bash
node scripts/build-kit.mjs   # one-time: unpack the Kenney zip → public/kit/* + pieces.json
npm install
npm run dev                  # http://localhost:5173
npm run build                # tsc -b && vite build  (always run before declaring done)
```
`public/kit/` is generated (committed for convenience). Source zip default: `/mnt/c/Users/laval/Downloads/kenney_building-kit.zip` (pass another path as arg).

## Architecture (the important part)

**Truth → derived.** The persisted truth is small; everything visible is derived.

- **Truth** (`store/useBuildStore.ts`, Zustand + persist): `cells` (`Record<"level,ci,cj", spaceId>`), `styleBySpace` (spaceId→style), `faceOverrides` (`Record<"level,ci,cj,dir", 'wall'|'window'|'door'>`), `roofOverrides`, `stairs` (`Stair[]`). Plus session state: `tool`, `activeLevel`, `selectedKeys`, `hoveredKey`, undo/redo snapshots.
- **Derive** (`kit/deriveSkin.ts`, **pure & deterministic**): `deriveSkin(cells, groundLvl, styleBySpace, faceOverrides, roofOverrides, fallback, stairs) → Instance[]`. Same input → same output (a stable `faceHash` chooses auto window/wall, no RNG/time). Re-runs on every edit inside `commit()`; the renderer/exporter just consume `instances`.
- **Render** (`three/`): `Scene` (Canvas, lights/sun, ground + grids, gizmo), `PlacedPieces`→`PieceInstance` (cloned GLB per instance via `useKitModel`), `GroundPlane` (pointer→cell, draw/erase + hover volume box), `exportGLB` (GLTFExporter, recenters X/Z).

### Coordinate model (don't break this)
A **cell is the square between grid lines**: cell (i,j) spans `[GRID*i, GRID*(i+1)]`, center `GRID*i+HALF`. Walls/columns sit **on the lines**; floors fill the squares. `GRID=2.0`, `WALL_HEIGHT=2.4` (= fixed floor height; stacked floors must match the wall height). `toCell = Math.floor(v/GRID)`. Constants in `kit/constants.ts`, all measured from the GLBs.

### Skinning rules (deriveSkin)
- Floor per occupied cell. Exterior wall on any face whose neighbour cell is empty (occupancy only — connected cells merge into one open building, no interior walls).
- Window/wall is a deterministic hash per face; one auto **entrance door** on the ground floor's front (−Z) face (single + hinged leaf, offset by `DOOR_LEAF_OFFSET`).
- `faceOverrides` let the user force wall/window/door per face. **Contiguous same-kind window/door faces auto-merge into wide (2-cell) pieces** (`wall-*-wide-square`) — width emerges from how many adjacent faces share a kind.
- Roof = `roof-flat-center` per top cell + `border` rail along the outer edge (continuous flat roof). `roofFallbackCenter` drops the rail.
- **Drawn roofs (pitched/round)** — the kit has flat roofs only, so shaped roofs are **procedural geometry**, an additive layer over the flat roof. Truth: `roofs: RoofRegion[]` (`{level, ci0..cj1 rect, style: 'gable'|'hip'|'dome'|'shed', rotated?}`). In the **roof tool** (屋顶), drag a rect over a rooftop → `addRoof` caps the building top inside the rect, trying the active level then the level just below (so drawing while standing on the rooftop **or** on the empty level above it both work), and stamps `activeRoofStyle`. Geometry is built in `three/roofGeometry.ts` (`roofGeometry(style,w,d,rise,rotated)` → BufferGeometry, triangle-soup for crisp facets; **hip on a square footprint collapses to a pyramid 攒尖**; dome = scaled half-ellipsoid; ridge/slope runs the longer axis, `rotated` flips it) and rendered by `three/ProceduralRoofs.tsx` **inside the export group** (so GLTFExporter serializes it). The drawn roof **keeps the kit's flat roof + `border` eave (cornice)** untouched and the pitched cap **sits on top of the cornice** (`y0 + ROOF_CORNICE`), oversailing by `ROOF_EAVE` — so the kit's eave detail shows and there's no z-fight with the walls. Sized to the footprint → 4×4 big, 3×3 small. **One roof per area**: `addRoof` replaces any existing roof it overlaps (same level, latest-wins). `commit` runs `pruneRoofs`, which keeps a roof only while its rect is a clean rooftop — building **on top of it** (a space drawn at `level+1`) or erasing its whole surface **invalidates and drops it** (draw a fresh one). deriveSkin is roof-agnostic (roofs are a pure render/export layer). Select a roof → dock chips change style / rotate / delete; `Delete`/`Backspace` removes it. `selectedRoofId` is mutually exclusive with `selectedKeys`/`selectedStairId`.
- Per-**space** `styleBySpace` (spaceId→style, not per-level) — a **structural/enclosure typology**, not a building program. Envelope is one of three kinds (`THEMES[style].walls` = `'full'|'low'|'none'`, ranked full>low>none): `enclosed` (full load-bearing walls + windows/doors; **corner** columns) / `semi` (post-and-slab **frame** + a waist-high perimeter parapet `wall-low`, balcony/loggia; **column grid**) / `open` (bare **frame**: no walls; **column grid**). An envelope piece is emitted on a face only when this cell's rank **exceeds** the neighbour's, and is this cell's kind (so enclosed-vs-semi → one full wall from the enclosed side, no double). The column grid is at `BAY` spacing (world-aligned, incl. interior posts ∪ convex corners) for frame styles; `constants.ts`, default 2 cells/4m. One level can mix styles.
- **Circulation** (vertical, `kit/deriveCirculation.ts`) — full design rationale + the stair-model facts + pitfalls in **`docs/circulation-system.md`** (read before touching stair geometry). Summary: stairs are **derived, not hand-placed** — `deriveCirculation(cells, {groundDoors}, circulation) → Stair[]` (deriveSkin computes the ground door faces). Two kinds of auto stair:
  - **Interior core** — per building (footprint-connected component), one vertical core, **solid model** (`stairs-closed`, set in `coreAt`), stacked per level (`coreAt` emits a flight only where a floor exists above → only built where there's an upper storey). Placement obeys a **HARD stair invariant** (`flightUsable`): a flight needs an **approach cell at its foot** `(ci−di,cj−dj)` and a **step-off at its top** (a floor neighbour of B on `level+1`) — both open floor, so a stair **never starts at the wall root** (the recurring model-misread bug; foot/top geometry recorded in `docs/circulation-system.md` §2). Among usable spots it's **scored, not door-anchored**: prefer the stair **tucked alongside a windowless wall, foot facing the room** (deriveSkin passes ground-floor opening faces `openFaces`). `seed`-chosen (re-roll = `seed++`). A footprint too small for *approach + 2-cell run* (e.g. 2×2) gets no auto core. (Manual stairs from `addStair`/rotate also default to `stairs-closed`.)
  - **Exterior stair = drawn PLATFORMS, chained by placement** (`expandPlatform(cells, platformKey, platformSet)`). In the stair tool you click empty outdoor cells → added to `circulation.platforms` ("level,ci,cj"); each platform tile is a landing (and `addPlatform` opens a door on the adjacent enclosed wall). A platform emits **one flight toward the next platform** if one sits **one floor down + exactly 3 cells away in a cardinal direction** — the flight model fills 2 sloped cells (neither flat), so the next platform must be the flat landing *beyond* them: `platform(0) · flight-high(1) · flight-low(2) · platform(3)`. **Direction follows the user's placement** — place the next platform LEFT vs RIGHT to fold back (switchback) vs go straight; the stair breaks at each platform (each is a landing). A platform with **no platform below** descends to the **nearest walkable surface — a building roof/terrace, or the ground** (whichever the run reaches first; `descendFlights`). **Building roofs are platforms too**, so a setback's roof is a valid landing and the stair stops on the terrace instead of being forced to the ground; auto-direction **prefers a terrace landing**, then the shortest run (`pickDescent`). All flights stay OUTSIDE the footprint (never cross a wall) — the only cell a flight may overlap is the roof it lands on. **Per-platform overrides** (select the tower in the select tool): `circulation.platformModel[key]` swaps the stair model (`stairs-open`/`-center`/`-closed`, all 2-cell drop-ins; `stairs-sides` is 3-cell, deferred), and `circulation.platformDir[key]` ("rotate") forces which edge a *straight* descent goes down (ignored if that direction isn't clear; chained towers ignore it — placement decides). **Hard invariant: every flight stays OUTSIDE the footprint — never crosses a wall** (both tangents blocked → platform only, no stair). No central well, no floating mid-air platforms — the only platform is the one you drew at the door. `addPlatform` also opens a `door` on the adjacent enclosed wall face (`platformDoorFace`). Flights share `stairKey = platform:<key>` (delete removes the group). (Stair model `stairs-open` measured from the GLB: pure diagonal, full-height step at the +Z edge, no built-in landings.)
  - **Interior vs exterior is by source**: interior auto-cores (`auto:*`/manual) cut `floorHoles` on `level+1`; **`platform:*` stairs never cut a hole** — even when a roof landing puts their bottom cell inside the footprint they sit ON the terrace, not through it (deriveSkin's hole loop skips `platform:` ids).
  - Overrides: `circulation.manual[]` (user-drawn stairs, kept) + `circulation.suppressed[]` (deleted auto cores, by `auto:…` id) + `circulation.platforms[]` (drawn exterior landings, by `"level,ci,cj"`; delete a flight removes the whole group via its `platform:<key>` stairKey); `circulation.auto` toggles the interior auto layer. A `Stair = {level, ci, cj, dir}` is one flight (`stairs-open`, 2 cells deep, high end faces `dir`). Additive — never changes wall/column/roof/style logic.

### Interaction (fixed Tinkercad convention — never swaps)
Right-drag = orbit, middle-drag = pan, wheel = zoom (all tools). Left = the active tool: **select** (default; click a wall face *or a stair* to select — faces: Shift to multi-select + arrow keys cycle the style; a selected stair shows rotate/delete chips, `Delete`/`Backspace` removes it) / **space** (drag a rect to fill cells) / **stair** (circulation: auto-places interior stair cores at doors — dock has 自动 toggle + 换一个/re-roll; click an empty outdoor cell to drop an exterior landing platform that auto-grows a stair down) / **roof** (屋顶: drag a rect over a rooftop → a procedural gable/hip/dome/shed roof, style chips in the dock; select a roof to restyle/rotate/delete) / **erase** (area eraser — wipes everything anchored in the dragged rect on the active level: cells, manual stairs, **exterior platform stairs** (landing cell in rect), and **drawn roofs** (rect overlaps the roof at this level, or the level below when erasing the empty floor above it — mirrors drawing a roof from above); auto interior stairs re-derive). Selection state: `selectedKeys` (faces), `selectedStairId` (one stair), and `selectedRoofId` (one roof) are mutually exclusive. `OrbitControls keyEvents={false}` so arrows don't pan. The blue per-level **drawing grid** (floating at the active level) shows **only under a drawing tool** — it's hidden in select mode (it clutters picking; selection is per-object, not per-level); the faint ground baseline grid stays. Shortcuts: V/S/T/E tools; **↑/↓ = floor up/down** (`stepLevel`: ↑ can reach one floor above the topmost built level, then stops; empty scene stays at 0); **←/→ = cycle the selected face's style**; `[`/`]` also change level.

**Exterior stairs** are drawn, not auto: in the **stair tool**, go to the floor with the door, click the empty outdoor cell just outside it → that drops a **landing platform**, and a wall-parallel stair auto-descends from it to the nearest walkable surface — a building roof/terrace or the ground (see Skinning rules → Circulation). You control the platform position exactly; the stair falls out of it.

## Verification
**Logic tests are the default** (bundle a `kit/*.ts` with `npx esbuild --bundle` and assert piece counts/positions in node) — cheap and deterministic. **Screenshots are the single most expensive step (image tokens); take at most one, and only when geometry/orientation genuinely can't be confirmed from logic.** A headless harness exists (Playwright + chromium swiftshader, store exposed on `window.useBuildStore` in dev). See `[[reference-playwright-harness]]` memory. Always `npm run build` before finishing.

**Working cadence (cost):** do **one feature per session** — context grows fast and every turn re-reads it, so a fresh session per feature keeps each turn cheap (lean on this CLAUDE.md + memory to get back up to speed). See `[[feedback-verification-cost]]`.

## Gotchas
- three 0.180 OrbitControls dolly methods are private (`_dollyIn`); the nav zoom in `Scene`'s `CameraBridge` scales `camera.position` instead.
- GLTFExporter doesn't keep instancing; fine at school scale.
- Don't reintroduce a hand/`grab` cursor (misleading — camera isn't on left).
- The auto **flat** roof has no per-cell style switch (kit ships flat tiles only). Shaped roofs (gable/hip/dome/shed) are a separate **drawn, procedural** layer — see the drawn-roofs bullet under Skinning rules; they are generated three.js geometry, not GLB pieces.
